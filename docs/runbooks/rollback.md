# Инструкция: откат и ручное переключение слота

Дата: 2026-04-20
Применимость: `TARGET_FORMAT=rauc`, `TARGET_PLATFORM=pc-efi`.

## Уровни отката

1. **Автоматический** — EFI BootNext + упавшая проверка работоспособности
   вернут к предыдущему слоту на следующую перезагрузку. Это работает
   "из коробки" после успешной установки новой версии, если проверка упала.
2. **Блокирующий** — администратор снимает плохой RAUC-пакет со `stable`-канала,
   новые панели перестают его получать.
3. **Ручное переключение слота** — администратор принудительно переключает активный слот
   через `rauc status mark-active` на конкретной панели.
4. **Жёсткий откат** — откат через установку предыдущего RAUC-пакета
   (с `force_downgrade=true` на сервере).

## Уровень 1: автоматический

Ничего делать не нужно. Описано в `docs/runbooks/qemu-pc-efi-test.md`
шаг 5.5 и `docs/runbooks/watchdog.md`.

Критерий успеха: `cat /etc/inauto/firmware-version` после второй
перезагрузки = предыдущая версия, `rauc status` показывает прежний слот =
`booted`, `good`.

## Уровень 2: снятие со stable-канала

Если плохая версия уже переведена в `stable` и часть парка её
подхватила:

```
# На сервере обновлений (один SQL-запрос):
sqlite3 /var/lib/.../server.sqlite3 \
  "UPDATE bundles SET channel='candidate' \
   WHERE compatible='inauto-panel-ubuntu-amd64-pc-efi-v1' \
     AND version='<BAD_VERSION>';"
```

Эффект:
- Необновившиеся панели `/api/latest` больше не вернёт BAD_VERSION,
  они останутся на предыдущей исправной версии.
- Уже обновившиеся панели, у которых проверка работоспособности упала, откатятся
  автоматически (уровень 1).
- Уже обновившиеся панели, у которых проверка работоспособности прошла, но проблема
  выявилась позже — см. уровень 3 или 4.

## Уровень 3: ручное переключение слота на конкретной панели

Если панель загрузилась в плохой слот (проверка работоспособности прошла, но потом
обнаружилась регрессия) — явно переключаем обратно:

```bash
# На панели:
sudo -i

rauc status
# Смотрим, какой слот загружен, а какой считается other.

# Переключаем BootOrder так, чтобы следующая загрузка пошла в other slot.
# На EFI-бэкенде:
rauc status mark-active other

# Перезагрузка, и следующая загрузка будет уже из прежнего слота
systemctl reboot
```

После перезагрузки:

```bash
cat /etc/inauto/firmware-version
# Должна быть версия прежнего слота.

rauc status
# Прошлый слот = booted; текущий (то есть бывший "новый") = active или bad.
```

**Замечание:** `rauc status mark-active other` на EFI-бэкенде
обновляет BootOrder (через `efibootmgr`). Нужны root-права и включённый
UEFI runtime (`/sys/firmware/efi`).

## Уровень 4: жёсткий откат через force_downgrade

Используется только в крайних случаях: плохая версия прошла в stable,
много панелей уже на ней, откат через BootNext не работает
(проверка работоспособности прошла несмотря на регрессию).

### Шаг 1 — Загрузить предыдущую исправную версию в `stable`

Если в БД `bundles` уже есть предыдущая исправная версия, она всё ещё
доступна. Убедимся что она в `stable`:

```
sqlite3 ... "SELECT version, channel FROM bundles \
             WHERE compatible='inauto-panel-ubuntu-amd64-pc-efi-v1' \
             ORDER BY version DESC LIMIT 5;"
```

### Шаг 2 — Временно исправить сервер

В MVP сервера обновлений нет готового переключателя `force_downgrade=true`
по RAUC-пакету, есть только поле в ответе `/api/latest`. Надо исправить
серверную часть:

```python
# В app/main.py, в api_latest():
#   ... если bundle.version == KNOWN_ROLLBACK_VERSION:
#       return JSONResponse({..., "force_downgrade": True})
```

Или заменить все stable'ы:

```
# SQL-способ: снять плохую версию, вернуть исправную в stable
UPDATE bundles SET channel='candidate' WHERE version='<BAD>';
# (исправная версия уже в stable, ничего менять не надо)
```

И вручную дать серверу знать, что это ответ с понижением версии.
**MVP-способ:** временный отдельный endpoint, подложенный на сервере на время
инцидента.

### Шаг 3 — Обновить панели

`panel-check-updates.timer` сработает у всех панелей в пределах
часа + RandomizedDelaySec. С `force_downgrade=true` агент установит
старую версию поверх новой.

### Шаг 4 — Убрать force_downgrade

Когда все панели вернулись — откатить правку сервера, чтобы следующие
обновления случайно не понижали версию.

## Осмотр состояния всего парка

```
sqlite3 ... <<'SQL'
SELECT
  serial,
  compatible,
  current_version,
  current_slot,
  last_error,
  last_seen
FROM panels
ORDER BY last_seen DESC;
SQL
```

Полезные вопросы:
- «Какие панели на BAD_VERSION?» — `WHERE current_version='<BAD>'`.
- «Кто давно не отвечал?» — `WHERE last_seen < datetime('now','-1 hour')`.
- «У кого last_error не NULL?» — `WHERE last_error IS NOT NULL`.

## Чего делать нельзя

- **Править GPT вручную после заводской подготовки.** RAUC и
  `panel-check-updates` ожидают partlabels — любое отклонение ломает
  и установку, и откат.
- **Очищать persist-раздел для "чистого сброса"** — потеряете SSH host keys,
  NetworkManager, `/etc/inauto/*`. Лучше переустановить через заводскую инструкцию.
- **Использовать `rauc install` на том же слоте, который загружен.** RAUC
  откажет, но если обойти — получите повреждённый rootfs текущего слота.
  `rauc install` всегда пишет в **other** slot.
